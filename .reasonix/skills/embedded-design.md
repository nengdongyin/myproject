## 设计方法论

### 第一原则：接口优先

在写任何实现代码之前，必须先完成接口设计。一个好的接口：

1. **完备** — 覆盖所有使用场景，调用者不需要"绕过接口"直接操作内部数据
2. **最小化** — 只暴露必须暴露的，宁少勿多。接口一旦发布就是负债
3. **语义清晰** — 看函数名 + 参数名就知道做什么，不需要读实现
4. **可错** — 所有可能失败的操作必须有错误返回路径
5. **幂等倾向** — `init/deinit` 可重复调用不崩溃，`close` 未 `open` 的句柄是空操作

### 第二原则：分层必严

嵌入式 C 项目的标准分层（自底向上）：

```
┌─────────────────────────────────────────┐
│  应用层 (Application)                    │  ← 业务逻辑，可替换
├─────────────────────────────────────────┤
│  中间件层 (Middleware / Protocol)        │  ← 协议栈、算法库
├─────────────────────────────────────────┤
│  功能模块层 (Service / Manager)          │  ← 参数管理、日志、诊断
├─────────────────────────────────────────┤
│  硬件抽象层 (HAL / PAL)                  │  ← 统一硬件接口，隔离平台差异
├─────────────────────────────────────────┤
│  驱动层 (Driver)                         │  ← 外设驱动，直接操作寄存器
├─────────────────────────────────────────┤
│  平台层 (BSP / CMSIS)                    │  ← 芯片厂商提供
└─────────────────────────────────────────┘
```

**铁的规则**：
- 上层只能调用直接下层，**禁止跨层调用**
- 下层绝不依赖上层（`#include` 方向与调用方向一致：上层 include 下层）
- 同层模块之间通过上层协调，**禁止横向依赖**（除非通过回调/事件总线解耦）
- 全局变量只在最底层（BSP/平台层）出现，向上必须封装

### 第三原则：资源可视

每一个设计决策必须附带资源预算：

| 资源 | 预算 | 说明 |
|------|------|------|
| RAM（静态） | N 字节 | 全局/静态变量、缓冲池、查找表 |
| RAM（动态峰值） | M 字节 | 栈深度估算 + 堆分配峰值 |
| Flash（.text） | K 字节 | 代码体积估算 |
| Flash（.rodata） | J 字节 | 常量/字符串/配置表 |
| CPU（初始化） | T ms | 启动初始化耗时 |
| CPU（热路径） | U μs | 最高频调用路径的耗时上限 |

如果某项不可接受，**在接口设计阶段就调整**，不等到实现阶段才发现。

## 嵌入式 C 编码规范

### 命名规范

| 元素 | 规则 | 示例 |
|------|------|------|
| 模块前缀 | 全大写 + 下划线，≤15 字符 | `PM_`, `FLASHDB_`, `UART_` |
| 函数名 | 小写 + 下划线，`模块前缀_动作_对象` | `pm_param_read()`, `flashdb_kvdb_init()` |
| 结构体/枚举类型 | 小写 + 下划线 + `_t` 后缀 | `pm_param_t`, `flashdb_ctx_t` |
| 宏/常量 | 全大写 + 下划线 | `PM_MAX_PARAM_COUNT` |
| 私有函数/变量 | `static` + 不加模块前缀（仅文件作用域） | `static int find_slot(key_t key)` |
| 不透明指针 | 类型名 + `_t*`，结构体定义留在 .c | `typedef struct pm_ctx_t pm_ctx_t;` (在 .h) |

### 文件组织

```
module_name/
├── module_name.h          # 公共接口（API）+ 公共类型定义
├── module_name_internal.h # 内部类型/共享宏（仅模块内 .c 引用）
├── module_name.c          # 核心实现
├── module_name_config.h   # 编译期配置（所有可调参数）
└── module_name_port.h     # 平台移植接口（如果需要跨平台）
```

### 防御性编程

```c
// 1. 参数校验：对外接口必须校验，内部 static 函数可用 assert
int pm_param_read(pm_ctx_t *ctx, key_t key, void *buf, size_t *len) {
    if (ctx == NULL || buf == NULL || len == NULL) {
        return PM_ERR_INVALID_ARG;
    }
    // ...
}

// 2. 返回值必须检查，不允许 (void) 丢弃
int rc = flashdb_kvdb_init(&kvdb);
if (rc != FLASHDB_OK) {
    return PM_ERR_STORAGE_INIT;
}

// 3. 固定大小的缓冲区必须用 sizeof 而非魔数
uint8_t buf[PM_BUF_SIZE];
memset(buf, 0, sizeof(buf));  // 不是 memset(buf, 0, 64);

// 4. 枚举 switch 必须覆盖所有值，缺少 default 让编译器警告未覆盖分支
switch (state) {
    case STATE_IDLE:    return "空闲";
    case STATE_ACTIVE:  return "活跃";
    case STATE_ERROR:   return "错误";
    /* 没有 default — 新增枚举值编译告警会提醒 */
}
return "未知";  // 防御性兜底
```

### 内存管理策略（按约束等级选择）

| 等级 | 策略 | 适用场景 |
|------|------|---------|
| **S0** | 全静态分配，禁止 `malloc` | FPGA 软核、安全关键系统 |
| **S1** | 静态池 + 编译期配置大小 | 大多数嵌入式项目 |
| **S2** | 静态池 + 堆（仅初始化阶段使用） | 需要运行时灵活配置 |
| **S3** | 全堆分配 + 内存池 | Linux 嵌入式、应用处理器 |

默认采用 **S1**，除非用户明确要求其他。

## 设计模式目录（嵌入式适用）

根据场景选择，**不是越多越好**：

| 模式 | 适用场景 | 嵌入式开销 | 示例 |
|------|---------|-----------|------|
| **不透明指针 (Opaque Pointer)** | 隐藏实现细节，几乎零成本 | RAM: 4B 指针 | `typedef struct pm_ctx_t pm_ctx_t;` |
| **工厂模式 (Factory)** | 需要运行时选择驱动/后端 | RAM: 静态函数表 | `flashdb_get_driver(type)` |
| **观察者/回调 (Observer)** | 事件通知、参数变更通知 | RAM: 回调指针数组 | `on_param_changed()` |
| **状态机 (State Machine)** | 协议解析、流程控制 | Flash: 状态表 | 函数指针表驱动或 switch 驱动 |
| **命令模式 (Command)** | 任务队列、日志缓冲 | RAM: 命令结构体数组 | 环形缓冲区 + 函数指针 |
| **策略模式 (Strategy)** | 可替换的算法（如校验/加密/压缩） | RAM: 函数指针 | `typedef int (*checksum_fn_t)(...);` |
| **单例 (Singleton)** | 硬件资源（UART/I2C/Flash 控制器） | 零额外开销 | 全局静态实例 |
| **双缓冲 (Double Buffer)** | DMA 传输、显示帧缓冲 | RAM: 2× buffer_size | ping-pong buffer |
| **环形缓冲 (Ring Buffer)** | 数据流、日志、命令队列 | RAM: buffer + 读写指针 | 无锁单生产者-单消费者 |
| **对象池 (Object Pool)** | 频繁分配释放的固定大小对象 | RAM: 静态池 | Free-list |

## 工作流程

### 第 1 步：参考材料 + 需求与约束收集
- **1a. 搜集参考材料**（按上述 P0→P1→P2 优先级执行）
  - 执行 `directory_tree docs/ref/` 确认是否存在参考目录
  - 若存在 `docs/ref/coding-style.md`，读取并以其规范为准
  - 若存在与设计相关的器件手册/协议文档，读取并以其中数据为准
  - 若项目内资料不足，用 `web_search` + `web_fetch` 补充
- **1b. 阅读用户需求** — 解析 `arguments` 中的功能描述、约束、参考代码
- **1c. 探索现有架构** — 如果涉及对接已有模块，用 `search_content` + `read_file` 理解现有接口
- **1d. 梳理分层结构** — 用 `directory_tree` 了解项目现有分层，确定新模块的插入位置
- **1e. 明确平台约束** — 如果用户没给，基于项目现有代码推断（如 `_config.h` 中的宏）

### 第 2 步：接口设计（最重要）
输出**完整的 .h 文件草案**（先不写实现），包含：
- 模块上下文类型（不透明指针声明）
- 所有公共 API 函数签名 + 完整 Doxygen 注释
- 错误码枚举
- 配置宏的默认值
- 回调函数类型定义

**接口设计检查清单**：
- [ ] 调用者能否仅凭 .h 文件完成集成？（不需要看 .c）
- [ ] 每个可能失败的操作是否都有错误返回路径？
- [ ] 是否有 `init/deinit` 配对？
- [ ] 是否存在"顺序陷阱"（必须先调 A 再调 B，但 A/B 的函数名没体现）？
- [ ] 参数是否 `const` 正确？（只读参数必须 `const`）
- [ ] 大小参数是否使用 `size_t` 而非 `int`？

### 第 3 步：分层规划
- 画出该模块在上述标准分层中的位置
- 列出直接依赖（下层）和被依赖（上层调用者）
- 标注数据流向

### 第 4 步：资源预算
- 按上述模板填写资源预算表
- 如果超出平台约束，在此步骤调整设计（缩缓冲、砍功能、改算法）

### 第 5 步：设计模式选择
- 从设计模式目录中选择适用的模式
- 说明选择理由和预期收益
- 如果有多个备选方案，给出对比

### 第 6 步：生成骨架代码
- 创建完整的 `.h` 文件（含 Doxygen 注释）
- 创建 `.c` 文件（函数骨架 + 实现注释标记 `/** TODO: 实现... */`）
- 创建 `_config.h`（所有可配置宏 + 说明）
- 如果涉及跨平台，创建 `_port.h`

### 第 7 步：输出设计文档
按下方格式输出完整的设计文档。

## 规则

- **接口先行** — 设计文档和 .h 文件必须在 .c 骨架之前完成
- **中文文档 + 中文 Doxygen** — 设计文档和注释均使用中文
- **资源可量化** — 每个设计决策必须附带资源影响说明，不允许"应该够用"
- **约束诚实** — 如果用户给出的需求在给定平台约束下不可行，必须明确指出并给出替代方案
- **生成可编译骨架** — .h 和 .c 骨架必须语法正确、可直接编译（函数体留空 + TODO 注释不算编译错误）
- **不跨层** — 生成的头文件 `#include` 方向必须自顶向下，不允许反向依赖
